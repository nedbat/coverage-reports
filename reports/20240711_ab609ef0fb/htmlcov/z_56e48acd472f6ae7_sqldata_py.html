128 """Manages collected coverage data, including file storage. 

129 

130 This class is the public supported API to the data that coverage.py 

131 collects during program execution. It includes information about what code 

132 was executed. It does not include information from the analysis phase, to 

133 determine what lines could have been executed, or what lines were not

135 

136 .. note:: 

137 

138 The data file is currently a SQLite database file, with a 

139 :ref:`documented schema <dbschema>`. The schema is subject to change 

140 though, so be careful about querying it directly. Use this API if you 

141 can to isolate yourself from changes. 

142 

143 There are a number of kinds of data that can be collected: 

144 

145 * **lines**: the line numbers of source lines that were executed. 

146 These are always available. 

147 

148 * **arcs**: pairs of source and destination line numbers for transitions 

149 between source lines. These are only available if branch coverage was 

150 used. 

151 

152 * **file tracer names**: the module names of the file tracer plugins that 

153 handled each file in the data. 

154 

155 Lines, arcs, and file tracer names are stored for each source file. File 

156 names in this API are case-sensitive, even on platforms with 

157 case-insensitive file systems. 

158 

159 A data file either stores lines, or arcs, but not both. 

160 

161 A data file is associated with the data when the :class:`CoverageData` 

162 is created, using the parameters `basename`, `suffix`, and `no_disk`. The 

163 base name can be queried with :meth:`base_filename`, and the actual file 

164 name being used is available from :meth:`data_filename`. 

165 

166 To read an existing coverage.py data file, use :meth:`read`. You can then 

167 access the line, arc, or file tracer data with :meth:`lines`, :meth:`arcs`, 

168 or :meth:`file_tracer`. 

169 

170 The :meth:`has_arcs` method indicates whether arc data is available. You 

171 can get a set of the files in the data with :meth:`measured_files`. As 

172 with most Python containers, you can determine if there is any data at all 

173 by using this object as a boolean value. 

174 

175 The contexts for each line in a file can be read with 

176 :meth:`contexts_by_lineno`. 

177 

178 To limit querying to certain contexts, use :meth:`set_query_context` or 

179 :meth:`set_query_contexts`. These will narrow the focus of subsequent 

180 :meth:`lines`, :meth:`arcs`, and :meth:`contexts_by_lineno` calls. The set 

181 of all measured context names can be retrieved with 

182 :meth:`measured_contexts`. 

183 

184 Most data files will be created by coverage.py itself, but you can use 

185 methods here to create data files if you like. The :meth:`add_lines`, 

186 :meth:`add_arcs`, and :meth:`add_file_tracers` methods add data, in ways 

187 that are convenient for coverage.py. 

188 

189 To record data for contexts, use :meth:`set_context` to set a context to 

190 be used for subsequent :meth:`add_lines` and :meth:`add_arcs` calls. 

191 

192 To add a source file without any measured data, use :meth:`touch_file`, 

193 or :meth:`touch_files` for a list of such files. 

194 

195 Write the data to its file with :meth:`write`. 

196 

197 You can clear the data in memory with :meth:`erase`. Data for specific 

198 files can be removed from the database with :meth:`purge_files`. 

199 

200 Two data collections can be combined by using :meth:`update` on one 

201 :class:`CoverageData`, passing it the other. 

202 

203 Data in a :class:`CoverageData` can be serialized and deserialized with 

204 :meth:`dumps` and :meth:`loads`. 

205 

206 The methods used during the coverage.py collection phase 

207 (:meth:`add_lines`, :meth:`add_arcs`, :meth:`set_context`, and
